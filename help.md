## 1. 이번에 겪은 구체 증상

### 1.1 동네 철물점 · 5천 원 (스크린샷, 2026-05-07)

**사용자 발화:**

```text
오늘 5000원 쓴 건 뭐 채워줘. 적요는 동네 철물점, 결제 카드 선택할게.
```

**화면 결과(요약):**

- 요약: `2026-05-07, 동네 철물점, ₩5,000. 항목을 선택하거나 직접 입력해 주세요.`
- **추천 칩:** `교육`, `운동/헬스` 조합처럼 **철물점·생활·쇼핑과 무관하게 느껴지는** 라벨.
- 체감: 필드 채워 주는 것처럼 보여도 **추천이 도움이 되지 않는다** → **AI가 일을 안 한다**로 인식.

### 1.2 축구클럽 월회비 등 (직전 회차)

- **식재료/마트·외식** 같은 식비형 칩 → 사용자 반발(“외식이라니?”).
- 수정 후에는 **교육·운동/헬스** 쪽 기계 순서 폴백이 앞으로 옴 → 회비류에는 나을 수 있으나 **일반 소매(철물점 등)**에는 여전히 맞지 않을 수 있음.

### 1.3 기타 UX

- **날짜:** 사용자가 말한 “오늘”과 화면 `YYYY-MM-DD`가 서버 `today` 기준이라 심리적으로 어긋난다고 느낄 수 있음(별도 검토).

## 2. 원인으로 추정되는 구조 (기술 메모)

1. **표시 풀 38+7은 “허용 라벨 목록”**이지, **항상 그 의미로 맞는 추천**을 보장하지 않음.
2. **`ensureCategoryCandidates` 3단계** 중 **기계 순서 폴백**은 **의미 추론 없이** 리스트 순서로 슬롯을 채움 → LLM·히스토리가 약하면 **엉뚱한 2개** 가능.
3. **`normalizeStructuredResult`의 풀 게이트**(지출/수입 추정)와 모델 출력이 안 맞으면 후보가 비었다가 폴백으로만 채워짐.
4. **저장 Enum**(10 지출·4 수입)과 **칩 문자열**(38·7)은 **`vaultStore` 매핑**으로만 접힘 — 칩이 이상해도 저장은 Enum으로 접힐 수 있음(사용자 체감과 분리).

## 3. 풀 38+7 새 세팅 — 무엇을 했는지 (상세)

외부 리서치(국가통계 축 · 앱/카드 분류 교차)를 반영한 **채팅 표시 전용 라벨**을 정본으로 넣었다. **카테고리 하드코딩(축구→OO)** 은 두지 않는 방향 유지.

### 3.1 서버 `netlify/functions/chat-assistant.js`

| 항목 | 내용 |
|------|------|
| `CHAT_DISPLAY_EXPENSE_POOL` | 지출 **38**개 (식재료/마트 … 멤버십) |
| `CHAT_DISPLAY_INCOME_POOL` | 수입 **7**개 (급여 … 환급/캐시백) |

| 동치 | `normalizeChipLabelToChatPool` — 구 Enum/구칩명만 풀 근처로 치환 |

## 4. 외부 조언 후 새 세션에서 다시 볼 것

1. **“빈 칩 금지” vs “엉터리 칩 금지”** — 현재 설계 트레이드오프를 제품 차원에서 다시 선택할지 (기계 폴백 축소·제거 등).
2. **LLM 재시도 / 신뢰도 임계값** 아래에서는 **칩 1개만** 또는 **칩 없이 직접 입력 안내만** 하는 UX.
3. **철물점·마트류**처럼 **상호 중심** 거래에서 **히스토리 매칭** 가중·`생활용품` 등 우선 서브풀 같은 **품질 개선**(거래별 키워드가 아니라 패턴 레벨 검토).
4. **`category_confirm` 응답 직전** 서버 로그 한 줄(`structured.category_candidates`, `guaranteedCandidates`)로 재현 고정.
